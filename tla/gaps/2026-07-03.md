# gh-audit gap report

**22 confirmed gaps out of 42 candidates. 17 are high severity. Worst class: false-compliant verdicts — unreviewed code marked as reviewed.**

> **Maintainer verification note (added post-run).**
> Gap #1 was checked by hand and is CONFIRMED at the mechanism level — it is the headline.
> - GitHub resolves a commit's `author.id` from the git author email.
> - Proof: email `181008794+kubernetes-prow[bot]@users.noreply.github.com` → API `author.id` = 181008794 (the id is just the email prefix).
> - This resolution is NOT gated on signing: an unsigned commit (`verified:false`) still gets `author.id` populated (e.g. k8s commit 4dabaeb).
> - The noreply email is free-text git config, so an attacker can set any account's id and push an unsigned commit.
> - gh-audit trusts this id at `client.go:1012` and never consults `IsVerified` for identity.
> - **Impact:** the "author_id is not forgeable" root-of-trust axiom holds for REVIEW/PR ids (from the account context) but is FALSE for COMMIT author ids on unsigned commits. That breaks §1 exempt and the §4 carve-out, and it is an explicit axiom in `Approval.tla` / `Exempt.tla` (`AtkPush` cannot pick the exempt id). Both specs need the adversary widened to choose any id on unsigned commits — TLC should then break `Sound`.
> The workflow's specific cited URL for #1 used a registered personal email, not a forged noreply address — weaker than claimed — but the underlying mechanism reproduces cleanly as above.

## The gaps

### 1. forged-exempt-author-email — high — cross-rule

**Scenario**
- PR approved on SHA A.
- Author pushes an unsigned commit on top with `--author 'ci-bot <ID+bot@users.noreply.github.com>'`.
- GitHub binds `author.id` to the exempt bot by email alone.
- §4 carve-out promotes the stale approval; the landing is marked compliant.
- Same trick grants the §1 exemption on a direct push.

**Why a gap**
- `isExemptCommit` (audit.go:233) and `isApprovalRefreshable` (audit.go:933) trust `AuthorID` alone.
- `commit.IsVerified` is only used for merge classification, never for exempt claims.
- Confirmed by a live push: an unsigned commit forged to id 22185274 resolved to that account with `verified:false`.
- `tla/Approval.tla:65` axiomatizes authorId as GitHub-bound — false for unsigned commits.

**Next step**
- Go test: unsigned post-approval commit forged to an exempt id → expect non-compliant.
- Code: gate exempt-authorship on `IsVerified` (or web-flow / API signature). Fail closed otherwise.
- TLA: widen `AtkPush` to allow `aid ∈ {Atk, NoId, Bot}` — TLC should then break `Sound`.
- Evidence: https://github.com/kubernetes/kubernetes/pull/140120/commits/4dabaebe45ddd8a01fb8253df329f6d176283a27

### 3. pr-commits-250-cap-truncates-squash-backstop — high — §1

**Scenario**

**Next step**

### 4. reverted-sha-not-shipped-history — high — §8

**Scenario**
- Attacker pushes commit A (exact inverse of their payload) to a throwaway branch.
- Then lands M on main: message `Revert "x"` + `This reverts commit <A>`, diff = the payload.
- `GetCommitFiles(A)` resolves; diffs are exact inverses → §8 waives M.
- Arbitrary NEW code ships as a "clean revert."

**Why a gap**
- No path checks that A is an ancestor of, or ever shipped on, an audited branch.
- `ResolveRevertedSHA` accepts the attacker-supplied trailer verbatim.
- §8 is the one waiver meant to resist a direct-push insider — audit.go:96 states it never inspects the reverted commit.
- `Revert.tla:66` hard-wires `diffVerified' = real` — only true if X shipped.

**Next step**
- Go test: revert pointing at a SHA never on any audited branch → expect non-compliant.
- Code: require RevertedSHA reachable in an audited branch's history before waiving.
- TLA: add a history variable so "X never shipped" is representable.

### 5. multiset-equality-is-not-exact-inverse — high — §8

**Scenario**
- Reviewed commit X adds `x = sanitize(x)` in handlerA; an identical line already exists in handlerB.
- Attacker's "revert" M removes the handlerB copy (same content, different hunk).
- `multisetEqual` sees one matching removed line → diff-verified → §8 waives.
- Result: handlerA keeps the unreviewed guard; handlerB LOST its sanitization.

**Why a gap**
- `IsCleanRevertDiff` (revert.go:179) compares per-file line multisets, dropping hunk offsets and context.
- Confirmed empirically: different-location removal diff-verifies.
- Trivial to trigger — any duplicated line (`return err`, `return nil`, a log line).
- `Revert.tla:52-53` defines diffVerified as byte-level exact inverse; the ship check is weaker.

**Next step**
- Go test: revert removes a byte-identical line from a different location → expect not clean.
- Code: compare parent-tree / blob SHAs, or keep hunk position in the multiset key.
- TLA/gaps: log that the implemented check is weaker than the modeled one.

### 6. reapply-escapes-prefix-gate — high — §8

**Scenario**
- Commit A lands. Reverted with a hand-worded subject `Revert upgrade to hash-for-dep 1.4.7` (no `Revert "..."` form).
- Later `git revert <that>` → subject `Revert "Revert upgrade to hash-for-dep 1.4.7"` + trailer.
- RevertOfRevert gate needs prefix `Revert "Revert "` → misses → classified ManualRevert.

**Why a gap**
- The chain guard lives only in `ParseRevert`'s message-prefix check (revert.go:57-61).
- `verifyRevertDiff` never asks whether the reverted target is itself a revert.
- Reproduced end-to-end with live ember.js data (message + inverse diffs verified via API).
- Deciding-rule variant on rust-lang/rust: bors approvals aren't GitHub reviews, so §4 fails and §8 alone flips the verdict.
- Contradicts the code's own comment (revert.go:29-30) and TODO.md.

**Next step**
- Go test: unquoted inner-revert subject → expect not-clean.
- Code: after ManualRevert diff-verification, fetch the reverted commit's message; refuse if it parses as any revert kind. Or gate on `Revert "Revert` prefix without the inner quote.
- Evidence: https://github.com/emberjs/ember.js/commit/cf2d358f7332d6b258d60fc3ccab3728b7d0781e (inner d8cef0d54375); rust-lang/rust 6c890bb9697c0971e70e1eba495b666516ecece6

### 9. merge-first-parent-diff-base — high — §2

**Scenario**
- Attacker branches feat off main, adds unreviewed commits.
- Mints M = `git commit-tree feat^{tree} -p feat -p main` and pushes M directly to main.
- M's tree equals its first parent, so `GET /commits/{M}` reports 0/0 and empty files.
- `applyEmptyCommitFallback` has no ParentCount gate → M stamped compliant "empty commit," though main's tree just changed by the whole feat diff.

**Why a gap**
- Stats are first-parent-relative; a foxtrot/crafted merge diverges from the audited branch's previous tip.
- Two variants stay fully green: silent revert (M reverts main's tree, zero flags) and foxtrotting a dev-approved branch onto main.
- Parent order is attacker-chosen — a forgeable input flips a verdict (violates commit 3c7e615).
- `EmptyCommit.tla:46` defines ground truth AS the stats, with no parent structure.

**Next step**
- Go test: ParentCount>1 with 0/0/0 stats → expect not waived as empty.
- Code: gate the empty-commit waiver on ParentCount==1, or verify the merge tree against the audited branch's prior tip.
- TLA: add parent structure so a Compliant-but-not-reallyEmpty state is reachable.

### 13. scope-set-divergence-sync-vs-reaudit — high — cross-rule

**Scenario**

### 17. post-merge-runs-unmodeled — high — §6

**Scenario**
- Admin bypass-merges a PR whose required check is red.
- Before gh-audit's first sync, someone re-runs the failed job and a flaky test passes.
- Sync captures both runs; latest-wins picks the post-merge green run → §6 success → the bypass disappears.
- Symmetric: a post-merge red re-run flips a legit-green merge to non-compliant on a fresh sync.

**Why a gap**

- Reviews DO get a merge-time cutoff (audit.go:606); §6 has no analogue.

- `Checks.tla:34` guards AddRun with ~merged — post-merge runs unreachable, so Sound is silent.

**Next step**
- Go test: red run at merge time + green post-merge re-run → expect non-compliant.
- Code: ignore check runs with completed_at after `pr.MergedAt`, mirroring the review cutoff.
- TLA: allow runs after Merge and re-prove Sound.

- ParseRevert sees ManualRevert (prefix `Revert "` but not `Revert "Revert "`); diff verifies against R → §8 waives.
- Variant: `Automatic revert of <R>..<old>` classifies AutoRevert with no textual clue. Known-vuln code re-lands compliant.

**Why a gap**
- The distinction lives only in ParseRevert's forgeable prefix match (revert.go:57).
- Nothing fetches R to detect the reverted SHA is itself a revert; no chain state.
- Violates the code's own invariant (revert.go:29-30) and commit 3c7e615.
- `Revert.tla`'s reallyReverts conflates "undoes shipped code" with "safe." TODO.md documents this as deferred.
- (Overlaps #6; this is the crafted-subject / AutoRevert framing.)

**Next step**
- Go test: crafted `Revert "cleanup"` + trailer to a revert R → expect non-compliant.
- Code: fetch R; if R is itself a revert, refuse the waiver regardless of message.
- TLA: add a chain dimension distinguishing "takes code off" from "brings code back."

### 21. null-completedat-inverts-latest — medium — §6

**Scenario**
- Check "CI" completes green at T1 on the head SHA.
- It's re-run; the re-run is in_progress when gh-audit syncs.
- GitHub returns completed_at=null → CompletedAt stays zero time.
- The latest-run selector compares CompletedAt first → the true latest (unconcluded) run sorts OLDEST → the stale green wins → §6 success.
- If the re-run later concludes FAILURE, the audit-once rule keeps the verdict compliant forever.

**Why a gap**
- `client.go:726-728` maps null completed_at to zero; the selector (audit.go:714-716) inverts the run's position from newest to oldest.
- The "unconcluded latest → missing" branch (audit.go:724-731) is unreachable for this input.
- `pipeline.go:70-73` audits a commit once, so the false verdict is never recomputed.
- `Checks.tla:36,75` forces every run to carry t in 0..MaxTime — an absent ordering key is unrepresentable.
- Not a forgeable-input flip: needs a pre-existing genuine green run; can't hide a red check.

**Next step**
- Go test: green concluded run + newer in-progress run (zero CompletedAt), both orders → expect missing/non-pass.
- Code: rank unconcluded runs (Status != completed) as NEWEST.
- TLA: represent a t-absent run.

### 22. merge-queue-sha-mismatch — medium — §6

**Scenario**
- Repo uses GitHub merge queue.
- Required checks run on the ephemeral merge-group commit (`refs/heads/gh-readonly-queue/...`).
- The PR head SHA carries few or none of the required runs.
- `evaluateRequiredChecks` filters to `cr.CommitSHA == pr.HeadSHA` → finds nothing → every merge-queued PR reports "missing" → the whole repo's §6 goes red.

**Why a gap**
- §6 is strictly head-SHA-scoped at fetch (caching.go:685), evaluation (audit.go:711), and the legacy status supplement.
- The audited commit's own SHA (the former merge-group commit carrying the runs) is never queried for check runs.
- `Checks.tla` models runs with no SHA field — the head/merge-group split is unrepresentable.
- Fail-closed only, but makes §6 unusable on merge-queue repos and pushes users to delete the rule.

**Next step**
- Go test / fixture: required run present only on the merge-group SHA → expect §6 satisfied.
- Code: also fetch check runs at `pr.MergeCommitSHA` (equals the merge-group commit post-merge).
- TLA: give runs a SHA and model two SHAs.

## Cross-cutting patterns

- **Forgeable identity** (#1, #2, #10, #11, #12): verdicts rest on client-set email, check-run name, status context, or an agent account — none carry non-forgeable provenance.
- **First-parent / ancestry blindness** (#9, #14, #16, #18): the code follows names or first-parent walks instead of proving the commit actually reached the audited branch by a reviewed path.
- **Revert waiver is too loose** (#4, #5, #6, #19, #20): §8 waives on diff-inverse alone, never checking history, position, file metadata, or revert chains.
- **Offline re-audit degrades data** (#7, #8, #13): the re-audit path loads fewer fields / different config than sync, flipping verdicts with no new evidence.
- **Spec models a single trusted stream** (#10, #11, #17, #21, #22): `Checks.tla` has no producer, no post-merge runs, no absent timestamp, no SHA — so TLC's Sound is silent on all four.
